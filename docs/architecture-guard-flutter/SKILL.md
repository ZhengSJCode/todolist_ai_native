# Architecture Guard — Flutter/Dart 架构健康度诊断

对 Flutter/Dart 代码按六大设计原则进行诊断，输出结构化报告：原则符合度评分 + 腐化风险清单 + 设计模式建议 + diff 形式重构建议。

---

## Hard Gate（强制门禁）

在任何 Flutter/Dart 代码生成或修改任务中，必须先完成六大原则自检，再输出最终代码。

- 不允许在未做原则检查的情况下直接提交实现代码
- 若存在 Critical（🔴）问题，默认先修复再交付
- 若因业务约束无法修复，必须显式说明权衡和后续治理计划
- 回复中应至少给出：触发原因、主要风险点、是否通过门禁

## 诊断流程（4步）

### Step 1：识别场景与输入类型

| 场景 | 典型输入 | 重点 |
|------|---------|------|
| **TDD** | 接口定义、类签名草稿、无实现的抽象类 | 接口分离、依赖注入、单一职责 |
| **功能实现** | 完整的实现代码，可能含 Widget/Service/Repository | 全六大原则，聚焦腐化风险 |
| **Code Review** | PR diff 或完整文件 | 全六大原则 + 是否引入新腐化点 |

若用户未说明场景，根据代码特征自动判断。

### Step 2：六大原则逐项诊断

加载 `references/solid-flutter.md` 获取 Flutter/Dart 专项判断规则，然后对每条原则评分：

| # | 原则 | 缩写 | 权重 | 核心检查点 |
|---|------|------|------|-----------|
| 1 | 单一职责原则 | SRP | 20% | 类/Widget 是否只有一个变化原因；UI 与业务逻辑是否混写 |
| 2 | 里氏替换原则 | LSP | 15% | 子类是否完整实现父类契约；override 是否破坏语义 |
| 3 | 依赖注入原则 | DIP | 20% | 是否依赖抽象而非具体实现；依赖是否可替换 |
| 4 | 接口分离原则 | ISP | 15% | 接口/抽象类是否粒度合适；是否存在"胖接口" |
| 5 | 迪米特原则 | LoD | 15% | 是否存在链式调用暴露内部结构；模块间耦合度 |
| 6 | 开闭原则 | OCP | 15% | 扩展新功能是否需要修改已有类；是否用继承替代修改 |

每条原则给出 **0–10 分**，并列出具体问题。

**综合健康度** = Σ(原则分 × 权重)，保留一位小数。

### Step 3：问题分级 + 设计模式建议

**问题分级：**

| 级别 | 标记 | 定义 |
|------|------|------|
| Critical | 🔴 | 严重腐化，影响可维护性或可扩展性，必须重构 |
| Warning  | 🟡 | 潜在腐化风险，建议在下次迭代前修复 |
| Suggestion | 🔵 | 可选优化，引入后提升架构质量 |
| Healthy | ✅ | 符合原则，值得保持的良好设计 |

**设计模式建议原则（防止过度设计）：**

仅在满足以下条件时才建议引入设计模式：
- 当前代码已出现 🔴 或多个 🟡 问题，且模式能直接解决
- 未来扩展需求明确，而不仅是"可能会"
- 引入成本（代码量/复杂度增加）低于长期维护成本

若不满足上述条件，明确注明"暂不建议引入模式，保持简单实现"。

加载 `references/flutter-patterns.md` 获取 Flutter 常用模式及适用场景。

### Step 4：按模板输出报告

严格按照 `references/output-template.md` 生成报告。

**重构建议规则：**
- 每条建议必须附 diff 代码片段
- diff 前用一句话说明**违反了哪条原则、为什么改**
- 保持最小改动原则，不重写整个文件
- TDD 场景下，diff 针对接口定义，不涉及实现细节

## 快速参考：Flutter 常见腐化信号

| 信号 | 违反原则 | 级别 |
|------|---------|------|
| Widget 内直接调用 `http.get()` / `Dio` | SRP + DIP | 🔴 |
| `StatefulWidget` 同时承担 UI + 业务逻辑 + 数据请求 | SRP | 🔴 |
| 子类 `override` 方法直接 `throw UnimplementedError()` | LSP | 🔴 |
| 依赖具体 Repository 实现而非抽象接口 | DIP | 🔴 |
| 一个 `abstract class` 有 10+ 个方法，实现类大量留空 | ISP | 🟡 |
| Widget 通过多层 `.child.data.user.name` 访问数据 | LoD | 🟡 |
| 新增功能靠在 `if-else` / `switch` 里加分支 | OCP | 🟡 |
| `BuildContext` 被传递到非 UI 层 | SRP + DIP | 🟡 |
| Provider/Bloc 直接持有具体 Service 实例 | DIP | 🟡 |
